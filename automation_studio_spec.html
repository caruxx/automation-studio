Automation Studio 仕様 & 統合計画

YouTube BGM チャンネル制作（SUNO 楽曲生成 → Premiere 配置 → 書き出し → YouTube 投稿）を単一 Web ダッシュボードで全自動化するツールの現状仕様の確定版と、重複機能の削減・共通設定の導入に向けた検討材料をまとめる。

対象: app.py 13,205行 / 263 API（→ D9 で 7,783行・4 router+app_core） 工程: 11（+plan=12） Python モジュール: 35本 チャンネル: orzz / SUKIMA ほか4 調査: 3エージェント並列コード監査
このドキュメントの位置づけ
既存の SPEC.md は変更履歴が 6/3 で止まり、工程数（6）も実態（11）と乖離している。本書は 実コードを正として現状を確定し直し、重複削減・共通設定・仕様確定の意思決定に使うことを目的とする。「要確認」は静的解析のみで運用実態が未確認の項目。

本書の分析を起点とした改善（確定プラン D1-D14）は大半が実装済み。本文中の app.py 13,205行 / 263ルート 等は分析時点の基準値であり、以下は実装で解消済み: D8(Flow/MJ 死蔵 ~2,400行撤去)・D6(competitor を取得層へ分離、分析は app_benchmark_analyze へ)・D9(app.py を 4 domain router + app_core に分割、12,649→7,783行。worker 基盤は runtime core として app.py 残置)・D10/D11/D12/D13/D5 他。未実施は D1/D3（無人稼働の開始＝要 GO）のみ。

1システム概要

AI 生成 BGM を編集・投稿する YouTube チャンネル運営を、3つの設計原則で最大限自動化する。

原則 1
API を使わない
Claude CLI を subprocess 起動、単一 JSON で往復。APIキー管理不要・認証ローカル完結
原則 2
ファイル＝真実
DB を持たず、動画フォルダ内の決まったファイル名の存在でステート管理
原則 3
1動画＝1フォルダ
{num}_{prefix}_{YYMMDD}。フォルダを見れば進捗が分かる

2アーキテクチャ

全ての操作経路が app.py（localhost:8888）という単一の頭脳に集約する構造。WEB UI・Claude/CLI・スケジューラの3経路が同じ FastAPI を叩く。

単一頭脳構造（3経路集約）
WEB UIindex.html SPA ─┐ Claude / CLIcurl・python3 app_pipeline.py ─┼→ app.py (:8888)FastAPI · 263 route ←─ APSchedulerschedule_jobs.json
3経路とも同じ app.py に集約 → ここが単一障害点であり、分割の最重要候補。✅D9実装済: domain API は routers/（benchmark/images/premiere_photoshop/youtube）へ、共通土台は app_core.py へ分離。worker 基盤（export/render queue・APScheduler）は runtime core として app.py 残置。
